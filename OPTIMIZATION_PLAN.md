# FlightSearch 產品優化計劃書

> 版本：v1.0
> 日期：2026-07-12（Asia/Taipei）
> 狀態：程式實作完成，待正式 Supabase migration 與部署驗收
> 範圍：保留 Cloudflare Pages、Zeabur FastAPI、Supabase、fast-flights、Kiwi MCP 與 LINE 通知；不做資料庫遷移、不加入會員系統、不使用 Superpowers 系列 skill。

---

## 1. 目標

把目前「可查多種機票組合」的網站，逐步提升為能協助個人判斷「現在是否值得買、外站票是否真的比較省」的工具。

完成後，使用者應能：

1. 分辨畫面上的價格是即時、快取或過期快取，以及資料來源是否近期正常。
2. 看到外站票加入定位票、行李、住宿與風險後的完整成本，而非只看主票價格。
3. 用較少的模式選擇完成一般找票或外站找票。
4. 在價格真正符合自己條件時，收到有理由的追蹤通知。

## 2. 已確認的現況與原則

| 項目 | 決定 |
|---|---|
| 資料庫 | 繼續使用 Supabase，不遷移 SQLite 或 Zeabur Volume |
| 前端與後端 | 維持 Next.js 靜態前端 + Cloudflare Pages Functions + Zeabur FastAPI |
| 資料來源 | 維持 fast-flights 主來源、Kiwi MCP 備援；不宣稱票價保證可訂 |
| 使用者帳號 | 維持匿名追蹤金鑰；先做恢復碼，不做登入／會員 |
| 開發範圍 | 每一批完成後均可部署與使用，不進行大規模重寫 |
| 優先判準 | 先提升資訊可信度，再提升外站票判斷能力，最後改善批次效率與進階分析 |

## 3. 優化路線圖

```text
第 1 批：資訊可信度
        ↓
第 2 批：外站票完整成本
        ↓
第 3 批：簡化成兩個使用入口
        ↓
第 4 批：有理由的價格追蹤
        ↓
第 5 批：批次掃描後端化
        ↓
第 6 批：歷史價格判斷與恢復碼
```

後一批可使用前一批累積的資料與規則；不可跳過第 1 批直接對使用者輸出「值得買」或「歷史低價」。

## 第 1 批：資訊可信度與可觀測性

### Task 1.1：建立真實健康狀態資料

**目的：** 讓首頁狀態燈反映近期實際狀態，而非僅代表 API 回應成功。

**工作內容：**

- 記錄每個 provider 最近一次成功／失敗時間、錯誤摘要、連續失敗次數與熔斷狀態。
- 記錄每日價格排程與追蹤排程最近一次執行結果。
- 擴充 `/api/health` 回傳內容，但不暴露 token、原始錯誤或個人追蹤資料。
- 首頁仍只顯示「正常／部分異常／異常」，詳細狀態放在可展開的診斷區。

**驗收條件：**

- [ ] 模擬 Kiwi 或 fast-flights 失敗後，健康 API 顯示對應 provider 的最近失敗資訊。
- [ ] Supabase 離線時，畫面顯示「部分異常」，並說明即時搜尋是否仍可用。
- [ ] 排程失敗時可在診斷區看見最近一次失敗時間。
- [ ] 健康 API 不回傳 secret、完整 upstream URL 或堆疊資訊。

**可能影響檔案：** `backend/main.py`、`backend/services/`、`backend/db/`、`frontend/src/lib/api.ts`、`frontend/src/hooks/useHealth.ts`、`frontend/src/components/SearchPage.tsx`。

**範圍：** M。

### Task 1.2：修正價格標籤與資料來源說明

**目的：** 避免把「本次搜尋中最低」誤認為「歷史低價」。

**工作內容：**

- 將外站掃描的「底價／一般／偏高」改為「本次最低／接近本次最低／高於本次最低」。
- 在結果卡標示資料來源、查詢時間、快取／過期快取與匯率換算資訊。
- 未達歷史資料門檻前，禁止顯示「歷史低價」或「現在最適合買」。

**驗收條件：**

- [ ] 同一次掃描的價格分級文字不含「歷史」或暗示長期比較的語意。
- [ ] 即時、快取、過期快取三種結果均有可辨識標示。
- [ ] 行動版 375px 不產生頁面水平捲動。

**可能影響檔案：** `frontend/src/lib/stationScan.ts`、`frontend/src/components/StationScanResults.tsx`、`frontend/src/components/FlightCard.tsx`、相關 E2E 測試。

## 第 2 批：外站票完整成本比較

### Task 2.1：定義外站票成本與風險規則

**目的：** 先確定「省錢」的計算方式，避免 UI 做完後才發現比較基準錯誤。

**工作內容：**

- 定義方案總成本：主要機票、定位票、必要住宿、行李、機場轉移與使用者自填預留費。
- 區分「已查到的票價」與「使用者估算成本」，不將估算值冒充即時報價。
- 定義風險旗標：分開開票、重新托運、跨機場、需過夜、轉機緩衝不足、可能 no-show 風險。
- 先以單程／單一出發日外站方案作為第一個可交付切片；來回與多城市之後再擴充。

**驗收條件：**

- [ ] 規則以測試案例固定：一般票 NT$ 32,800、外站主票 NT$ 19,500、定位票 NT$ 5,200、附加成本 NT$ 2,000 時，完整成本為 NT$ 26,700、節省 NT$ 6,100。
- [ ] 沒有定位票時，結果明確顯示「尚未計入定位票，不可判定實際節省」。
- [ ] 風險判斷不會因資料不足而顯示「低風險」。

**依賴：** Checkpoint 1。

**可能影響檔案：** 新增外站成本計算模組與單元測試、`frontend/src/lib/stationScan.ts`。

**範圍：** M。

### Task 2.2：外站方案完整成本結果卡

**目的：** 讓使用者直接看到外站方案是否真的較省、差額是否值得承擔風險。

**工作內容：**

- 在既有外站掃描結果加入「查看完整成本」動作。
- 顯示一般台灣出發、外站主票、定位票、附加成本、完整成本與預估節省。
- 顯示風險旗標和簡短白話說明；不提供法律、簽證或航空公司規則的保證。
- 預估節省不足使用者設定門檻時，標示「節省有限」。

**驗收條件：**

- [ ] 使用者可在不離開結果頁的情況下完成成本比較。
- [ ] 未查到定位票時不顯示「值得買」。
- [ ] 所有總額均可追溯到個別成本項目。
- [ ] 桌機與 375px 手機畫面可讀且不溢出。

**依賴：** Task 2.1。

**可能影響檔案：** `StationScanResults.tsx`、新元件／hook、`frontend/src/lib/api.ts`、E2E 測試。

**範圍：** M。

### Checkpoint 2：外站票第一個可用版本

- [ ] 外站主票低價但加上定位票後較貴的案例，正確顯示「不值得」。
- [ ] 已知完整成本較省的案例，正確顯示節省金額與風險。
- [ ] 前後端測試、build、兩種瀏覽器 E2E 全綠。

- 保留原有 multi、combo、scan 能力，改為外站流程的進階選項，不刪除既有分享連結格式。

- [ ] 原有 `mode=multi`、`mode=combo`、`mode=scan` 分享連結仍可開啟。
- [ ] 一般單程／來回搜尋流程不改變。

**依賴：** Checkpoint 2。

**可能影響檔案：** `SearchPage.tsx`、搜尋卡片與 hooks、分享連結 E2E。

## 第 4 批：有理由的價格追蹤與通知

### Task 4.1：擴充追蹤規則與通知理由

**目的：** 讓 LINE 通知說明「為什麼現在值得看」，避免只推送裸價格。

**工作內容：**

- 支援目標價、降價金額與符合外站完整成本門檻等規則。
- 依距離出發日調整檢查頻率，並對 provider 配額設總量上限與錯峰排程。
- 通知加上前次價格、降幅、目標價、資料時間及必要風險提示。

**驗收條件：**

- [ ] 價格低於目標價的通知含航線、目前價格、目標價與資料時間。
- [ ] 價格未變時不重複通知。
- [ ] 在配額上限情境下，排程會安全遞延並留下可查狀態。

**依賴：** Checkpoint 2。

**可能影響檔案：** `backend/services/tracker_service.py`、`scheduler.py`、`notifier.py`、schema migration、`TrackerDrawer.tsx`、測試。

**目的：** 將多日期、多外站的查詢從瀏覽器派工改為後端工作，讓使用者離開頁面後仍能完成。

**驗收條件：**

## 第 6 批：歷史價格判斷

### Task 6.1：以可解釋規則建立歷史比較

**目的：** 在資料樣本足夠時，才提供「近期低價／價格普通／偏高」判斷。

**工作內容：**

- 用 Supabase 價格歷史計算 30／60／90 天的最低、平均、中位數與樣本數。
- 至少有 10 筆有效且可比較樣本才輸出判斷；不同 provider 的資料需標示來源差異。
- 以透明規則產生文案，不做黑箱購買預測。

**驗收條件：**

- [ ] 樣本不足時顯示「資料累積中」，不給買／等建議。
- [ ] 價格低於近 90 天第 20 百分位時，顯示「近期低價」及樣本數。
- [ ] API 與 UI 能顯示計算期間與資料來源限制。

**依賴：** Checkpoint 1、持續累積歷史資料。

**可能影響檔案：** `backend/db/repository.py`、`main.py`、`PriceTrendChart.tsx`、API 型別、測試。

**範圍：** M。

### Checkpoint 6：決策輔助可解釋

- [ ] 所有判斷均能回溯到價格歷史與明確公式。
- [ ] 使用者能區分「本次最低」與「近期低價」。

## 5. 主要風險與處理方式

| 風險 | 影響 | 處理方式 |
|---|---|---|
| 第三方來源的價格短暫失效或不可訂 | 高 | 顯示資料時間與來源；一律導向訂票頁再次確認 |
| 外站票實際成本不完整 | 高 | 拆分即時票價與使用者估算；資訊不足時不得給「值得買」結論 |
| 批次查詢耗盡 provider 額度 | 高 | 後端排程、快取優先、併發／速率／總量上限 |
| 歷史資料不足卻做價格建議 | 中 | 設最低樣本數與「資料累積中」狀態 |
| 舊分享連結失效 | 中 | 保留現有 mode 參數，加入回歸 E2E |
| Supabase migration 版本不一致 | 中 | 每批 schema 改動建立可追蹤 migration、部署前檢查版本 |

## 6. 現階段不做事項

- 不移除或替換 Supabase。
- 不加入會員、付款、代訂票或自動購票。
- 不以機器學習預測未來票價。
- 不保證航空公司票規、簽證、入境資格或 no-show 結果。
- 不在未驗證第三方資料來源可穩定使用前，擴大掃描頻率或範圍。
